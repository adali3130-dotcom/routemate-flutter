## Project Overview

RouteMate is a vending machine route management system. This Flutter app replaces the
existing Kotlin Android driver app while keeping the Firebase backend completely unchanged.

**What this app is:**
- Driver-facing mobile app for Android (primary) and iOS
- Lets drivers view assigned visits, get directions, mark visits complete, submit timesheets,
  and update inventory

**What this app is NOT:**
- The web admin dashboard (React/Firebase Hosting — do not touch)
- The Firebase backend (Cloud Functions, Firestore rules — do not touch)

## Firebase Auth & Custom Claims

Every driver authenticates with email/password. After login, the Firebase Auth token
contains custom claims set by the `syncUserClaims` Cloud Function:

```
role        — "admin" or "driver"
company_id  — tenant identifier e.g. "VzHiN9O4lVLtkf3TuEaO"
```

### CRITICAL: How to read company_id

**ALWAYS force a token refresh before reading claims. NEVER read company_id from Firestore.**

```dart
final user = FirebaseAuth.instance.currentUser!;
final tokenResult = await user.getIdTokenResult(true); // true = force refresh
final companyId = tokenResult.claims?['company_id'] as String?;
```

Drivers do NOT have documents in a `users` or `drivers` Firestore collection that are
readable by the driver. The only reliable source of `company_id` is the auth token claim.

This was the root cause of the inventory bug in the Kotlin app — always use token claims.

## Firestore Collections & Schemas

### weekly_plans/{planId}
```
driver_uid    String   — Firebase Auth UID of assigned driver
driver_id     String   — driver email
week_start    String   — ISO date "YYYY-MM-DD", always a Monday
company_id    String   — tenant identifier
plan_id       String   — document ID
visits        List     — array of visit objects (see below)
```

**Visit object within visits[]:**
```
account_id    String   — references accounts collection
account_name  String   — display name
address       String   — full address (may be empty for old accounts)
date          String   — ISO date "YYYY-MM-DD"
day           String   — "Mon" | "Tue" | "Wed" | "Thu" | "Fri"
driver_id     String   — driver email
driver_uid    String   — driver UID
completed     bool     — completion status
notes         String   — visit notes
order         int      — route ordering integer
```

### timesheets/{driverEmail__weekStart}
Document ID format: `email@example.com__2024-01-15`

```
driver_email  String   — driver email
week_start    String   — ISO date "YYYY-MM-DD"
company_id    String   — tenant identifier
days          List     — array of daily entry objects (see below)
```

**Daily entry object within days[]:**
```
date          String   — ISO date "YYYY-MM-DD"
start_time    String   — "HH:MM"
end_time      String   — "HH:MM"
mileage       number   — kilometers driven
totalTime     String   — computed e.g. "2h 15m"
notes         String   — daily notes
```

**CRITICAL:** Timesheet data is stored in the `days` array on the PARENT document.
There is NO `dailyEntries` subcollection. This was a bug in the Kotlin app's report
generation — always read from `days` array on the `timesheets/{id}` document.

### accounts/{accountId}
```
account_name    String   — display name (use this, NOT "name")
address         String   — full address (empty on old records)
company_id      String   — tenant identifier
days_of_week    List     — array of day strings
frequency       String   — "weekly"|"twice_weekly"|"thrice_weekly"|"monthly"|"every_two_weeks"
```

**Account data quirk:**
- Old records (IDs a1–a65): empty addresses, may have stale data
- New records (IDs a101–a154): full addresses, correct days_of_week
- Always handle empty address fields gracefully — never crash on missing address

### inventory/{productId}
```
product_name        String     — display name
quantity            int        — current stock level
low_stock_threshold int        — threshold for low stock alert
company_id          String     — tenant identifier
updated_at          Timestamp  — last update time
updated_by          String     — email of last updater
notes               String     — procurement notes
```

### drivers/{driverUid}
```
name        String   — display name
email       String   — driver email
company_id  String   — tenant identifier
```

### location_requests/{requestId}
```
driver_email  String   — target driver email
status        String   — "pending" | "done"
```

### driver_locations/{driverEmail}
GPS location data written by driver app when responding to location requests.

## Routing & Auth Guard

GoRouter is configured with a redirect that checks auth state on every navigation:

```dart
redirect: (context, state) {
  final isLoggedIn = ref.read(authStateProvider).value != null;
  final isOnLogin = state.matchedLocation == '/login';
  if (!isLoggedIn && !isOnLogin) return '/login';
  if (isLoggedIn && isOnLogin) return '/plan';
  return null;
}
```

Routes:
- `/login` — LoginScreen
- `/plan` — WeeklyPlanScreen (default after login)
- `/timesheet` — TimesheetScreen
- `/inventory` — InventoryScreen

Bottom navigation bar switches between `/plan`, `/timesheet`, `/inventory`.

## Feature Specifications

### 1. Authentication
- Email/password login via Firebase Auth
- Auto-login: Firebase persists auth state across app restarts
- Forgot password: `FirebaseAuth.instance.sendPasswordResetEmail(email: email)`
- On login success: force token refresh immediately to pre-load company_id claim
- Auth state watched via `FirebaseAuth.instance.authStateChanges()` stream

### 2. Weekly Plan Screen

**Query logic:**
```dart
// Always query by driver_uid, not driver_id
// Try next week first, fall back to current week
final now = DateTime.now().toUtc(); // Use UTC to avoid timezone issues
final monday = now.subtract(Duration(days: now.weekday - 1));
final thisWeekStart = DateFormat('yyyy-MM-dd').format(monday);
final nextWeekStart = DateFormat('yyyy-MM-dd').format(monday.add(Duration(days: 7)));

// Try next week first
final nextWeekQuery = await firestore
    .collection('weekly_plans')
    .where('driver_uid', isEqualTo: uid)
    .where('week_start', isEqualTo: nextWeekStart)
    .get();

final weekStart = nextWeekQuery.docs.isNotEmpty ? nextWeekStart : thisWeekStart;
```

**Address enrichment (fix for Kotlin bug):**
At load time, for any visit where `address` is empty or blank:
```dart
final accountDoc = await firestore.collection('accounts').doc(visit.accountId).get();
final enrichedAddress = accountDoc.data()?['address'] as String? ?? '';
```

**Directions button:**
```dart
final uri = Uri.parse('https://www.google.com/maps/dir/?api=1&destination=${Uri.encodeComponent(address)}');
await launchUrl(uri, mode: LaunchMode.externalApplication);
```

**Mark complete:**
```dart
// Update the completed field within the visits array
// Use a Cloud Function or arrayRemove/arrayUnion pattern
// Simplest: update the entire visits array with the modified visit
```

**Day filter tabs:** Mon / Tue / Wed / Thu / Fri — filter visits by `day` field.

### 3. Timesheet Screen
- Driver selects a specific date
- Inputs: start_time (time picker), end_time (time picker), mileage (number), notes (text)
- Compute totalTime from start/end
- Save using `arrayUnion` on `days` field:
```dart
final docId = '${driverEmail}__${weekStart}';
await firestore.collection('timesheets').doc(docId).set({
  'driver_email': driverEmail,
  'week_start': weekStart,
  'company_id': companyId,
  'days': FieldValue.arrayUnion([dayEntry]),
}, SetOptions(merge: true));
```
- Submit email via Cloud Function `sendTimesheetEmail`
- Email must include: driver name, email, week_start, specific date, start_time, end_time,
  mileage, totalTime

### 4. Inventory Screen
- Force token refresh → read company_id claim (this is the fix for the Kotlin bug)
- Query: `firestore.collection('inventory').where('company_id', isEqualTo: companyId)`
- Sort: low stock items (quantity <= low_stock_threshold) at TOP, highlighted in red/amber
- Stepper buttons: +1 / -1 on quantity, save immediately on each tap
- Full Restock button: sets quantity to `low_stock_threshold * 4`
- Notes field: editable per product, saved on change
- Update `updated_at` (Timestamp) and `updated_by` (driver email) on every change

### 5. Location Sharing
- Listen to `location_requests` collection filtered by driver_email and status == "pending"
- On pending request: get GPS coordinates, write to `driver_locations/{driverEmail}`
- Update request status to "done"

## Known Issues & Their Fixes

| Issue | Root Cause | Fix |
|-------|-----------|-----|
| Inventory screen empty | company_id read from Firestore (no access) | Always use `getIdTokenResult(true)` + read from claims |
| Directions button no address | Visit address blank on old accounts | Enrich addresses from accounts collection at load time |
| Wrong week shown | Device timezone causing Monday calculation errors | Use UTC for all date calculations |
| Timesheet data missing in reports | Reading from non-existent dailyEntries subcollection | Always read from `days` array on parent timesheets document |

## Development Patterns

### Always use UTC for dates
```dart
final now = DateTime.now().toUtc();
```

### Token refresh pattern (use everywhere company_id is needed)
```dart
Future<String?> getCompanyId() async {
  final user = FirebaseAuth.instance.currentUser;
  if (user == null) return null;
  final tokenResult = await user.getIdTokenResult(true);
  return tokenResult.claims?['company_id'] as String?;
}
```

### Firestore error handling
Always wrap Firestore calls in try/catch. Show user-friendly error messages.
Never let FirebaseException bubble up to the UI uncaught.

### Loading states
Every async operation must show a loading indicator.
Use AsyncValue from Riverpod — handle loading/error/data states in UI.

### Naming conventions
- Files: snake_case
- Classes: PascalCase
- Variables/methods: camelCase
- Riverpod providers: camelCase ending in `Provider`
- Models use `fromMap(Map<String, dynamic>)` and `toMap()` methods
